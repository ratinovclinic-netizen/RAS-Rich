# План подключения ИИ-контролёра к Bitrix24 и Google Sheets

**Версия:** 0.1  
**Дата:** 21 июля 2026 года  
**Статус:** проектная схема до начала разработки интеграций

## 1. Принятое допущение

Под «док-таблицами» понимаются Google Sheets. Таблицы внутри обычных Google Docs не рекомендуется использовать как источник финансовых и операционных данных: у них нет удобной строковой модели, стабильных диапазонов и простого контроля схемы.

## 2. Цель интеграции

Подключить Bitrix24, Google Sheets и Telegram так, чтобы приложение:

- только читало бизнес-данные;
- не меняло сделки, задачи, воронки, документы и таблицы;
- сохраняло снимки исходных данных;
- проверяло источники на полноту и противоречия;
- рассчитывало KPI программными модулями;
- формировало вечернюю сводку в 20:00;
- предоставляло защищённый расширенный отчёт.

## 3. Общая схема

## 4. Подключение Bitrix24

### 4.1. Рекомендуемый порядок

#### Этап 1 — пилот на одном портале

Использовать входящий webhook Bitrix24, созданный от отдельного технического пользователя с минимальными правами чтения.

Преимущества:

- быстрое подключение;
- подходит для внутренней интеграции одного портала;
- не требует реализации обновления OAuth-токенов;
- достаточно для ежедневной и периодической выгрузки CRM.

Ограничение: секрет webhook не истекает автоматически, поэтому он должен храниться только на сервере и подлежать немедленной замене при подозрении на утечку.

#### Этап 2 — промышленная версия

При необходимости событий в реальном времени и более сложных возможностей использовать серверное локальное приложение Bitrix24 с OAuth 2.0 и подписками на события.

### 4.2. Защита режима «только чтение»

Один только REST-раздел `crm` не гарантирует запрет всех методов записи. Поэтому ограничение реализуется одновременно на нескольких уровнях:

1. отдельный технический пользователь Bitrix24;
2. CRM-роль пользователя только на просмотр необходимых сущностей;
3. минимальные разрешения webhook/приложения;
4. программный allowlist разрешённых REST-методов;
5. запрет методов с суффиксами `add`, `update`, `delete`, `set` и аналогичных;
6. журнал каждого запроса к Bitrix24;
7. автоматический тест, подтверждающий отсутствие операций записи;
8. хранение секрета только на сервере.

### 4.3. Методы первой версии

Предварительный allowlist:

- `crm.item.list` — универсальное чтение лидов, сделок и смарт-процессов;
- `crm.item.get` — детализация конкретного объекта;
- `crm.item.fields` — карта стандартных и пользовательских полей;
- `crm.type.list` — идентификаторы смарт-процессов;
- `crm.category.list` — направления/воронки;
- `crm.status.list` — стадии и справочники;
- `crm.activity.list` — звонки, встречи, задачи и другие CRM-активности;
- `crm.timeline.comment.list` — доступные комментарии таймлайна;
- методы чтения пользователей для сопоставления ответственных сотрудников.

Точный список уточняется после автоматической инвентаризации портала.

`crm.deal.list` можно встретить в старых интеграциях, но официальная документация рекомендует для новой разработки универсальный `crm.item.list`.

### 4.4. Минимальный набор полей сделки

- ID сделки;
- направление/воронка;
- стадия;
- сумма;
- валюта;
- ответственный;
- источник;
- дата создания;
- дата последнего изменения;
- дата закрытия;
- контакт/компания;
- признак успеха/проигрыша;
- причина проигрыша;
- пользовательские поля продукта, договора, оплаты и сопровождения;
- следующая активность и её срок;
- история изменения стадий при доступности.

### 4.5. Первичная инвентаризация Bitrix24

Первый технический запуск должен выгрузить:

1. список воронок;
2. список стадий каждой воронки;
3. список полей и пользовательских полей;
4. список пользователей и ролей;
5. ограниченную выборку сделок за один месяц;
6. активности ограниченной тестовой выборки;
7. доступные сведения о WhatsApp/открытых линиях.

По результату создаётся карта:

```text
Поле Bitrix24 -> Бизнес-смысл -> Тип -> Обязательность -> Правило проверки
```

## 5. WhatsApp через Bitrix24

WhatsApp подключается не напрямую, а через существующий канал/провайдера Bitrix24, если сообщения и активности сохраняются в CRM.

Перед разработкой анализа переписок проверяется:

- где именно Bitrix24 хранит историю канала;
- какие REST-методы доступны текущему тарифу и интеграции;
- сохраняется ли полный текст сообщений;
- доступны ли автор, время, направление и связь со сделкой;
- есть ли правовое основание для анализа коммуникаций;
- требуется ли обезличивание перед передачей в ИИ-модуль.

Если API не предоставляет полный текст, первая версия анализирует только доступные активности и показатели скорости реакции, не имитируя наличие переписки.

## 6. Подключение Google Sheets

### 6.1. Рекомендуемая авторизация

Создать отдельный Google Cloud-проект и сервисный аккаунт для приложения.

Для каждого нужного файла:

1. владелец таблицы открывает доступ адресу сервисного аккаунта;
2. выдаётся роль **Viewer / Читатель**;
3. приложение использует scope `https://www.googleapis.com/auth/spreadsheets.readonly`;
4. в конфигурации указываются ID таблицы и разрешённые диапазоны;
5. приложение читает данные через `spreadsheets.values.batchGet`.

Если сервер размещён в Google Cloud, предпочтительны короткоживущие учётные данные без постоянного JSON-ключа. Если применяется JSON-ключ, он хранится только в менеджере секретов и регулярно меняется.

### 6.2. Предлагаемая структура таблиц

#### Лист `Payments`

| Поле | Назначение |
|---|---|
| `payment_id` | Уникальный идентификатор платежа |
| `deal_id` | ID сделки Bitrix24 — основной ключ сверки |
| `payment_date` | Дата платежа |
| `amount` | Сумма в исходной валюте |
| `currency` | Код валюты |
| `status` | Получен / ожидается / возвращён / отменён |
| `source_reference` | Ссылка или номер первичного документа |
| `updated_at` | Время последнего обновления строки |

#### Лист `Contracts`

| Поле | Назначение |
|---|---|
| `contract_id` | Уникальный номер договора |
| `deal_id` | ID сделки Bitrix24 |
| `status` | Подготовка / согласование / подписание / подписан / отменён |
| `amount` | Сумма договора |
| `currency` | Валюта |
| `signed_at` | Дата подписания |
| `responsible_id` | Идентификатор ответственного юриста |
| `updated_at` | Время обновления |

#### Лист `PNL`

| Поле | Назначение |
|---|---|
| `period` | Отчётный период |
| `operation_date` | Дата операции |
| `direction` | Доход / расход |
| `category` | Статья P&L |
| `amount` | Сумма |
| `currency` | Валюта |
| `deal_id` | ID сделки, если применимо |
| `comment` | Проверяемое пояснение |
| `updated_at` | Время обновления |

#### Лист `Plans`

| Поле | Назначение |
|---|---|
| `period` | Месяц/год |
| `scope` | Отдел / команда / сотрудник |
| `employee_id` | ID сотрудника Bitrix24, если применимо |
| `metric` | Название планового показателя |
| `value` | Плановое значение |
| `currency` | Валюта, если применимо |

#### Лист `Support`

| Поле | Назначение |
|---|---|
| `case_id` | ID обращения или обязательства |
| `investor_id` | Внутренний обезличиваемый идентификатор |
| `deal_id` | Связанная сделка |
| `status` | Состояние |
| `created_at` | Дата создания |
| `due_at` | Крайний срок |
| `closed_at` | Дата закрытия |
| `responsible_id` | Ответственный |

### 6.3. Обязательное правило ключей

Во всех таблицах, связанных со сделками, должен присутствовать стабильный `deal_id` из Bitrix24. Сверка по имени инвестора, номеру телефона или текстовому названию сделки допускается только как вспомогательный поиск ошибки, но не как основной ключ.

## 7. Частота обновления

Предварительный регламент:

- Bitrix24: инкрементальная загрузка каждые 15 минут;
- Google Sheets: загрузка утверждённых диапазонов каждые 30 минут;
- контрольный вечерний срез: 19:45 Asia/Bishkek;
- проверка и расчёты: 19:45–19:55;
- Telegram-сводка и расширенный отчёт: 20:00;
- критические события: после очередной успешной загрузки;
- полная контрольная сверка истории: один раз в сутки ночью.

Частота хранится в конфигурации.

## 8. Хранение и обработка

Для каждой загрузки сохраняются:

- источник;
- время запроса;
- время данных в источнике;
- технический ID загрузки;
- исходный ответ или его защищённый снимок;
- контрольная сумма;
- результат нормализации;
- ошибки качества;
- версия правил расчёта.

Основные слои данных:

1. `raw` — неизменяемые исходные снимки;
2. `normalized` — единая модель сделок, платежей, договоров и сотрудников;
3. `analytics` — рассчитанные KPI и прогнозы;
4. `reports` — версии отправленных отчётов и доказательств.

## 9. Сверка источников

Минимальные автоматические проверки:

- `Bitrix deal_id` найден в платежах и договорах;
- сумма сделки совпадает с договорной суммой либо отклонение объяснено;
- платёж не превышает договор без допустимого правила;
- оплаченная сделка имеет допустимый юридический статус;
- валюта присутствует и допустима;
- даты имеют правильную последовательность;
- итог листа P&L равен сумме детализации;
- активная сделка имеет ответственного и следующее действие;
- отсутствуют дубли `payment_id` и `contract_id`;
- источник был обновлён до контрольного среза.

При конфликте агент:

1. не исправляет источник;
2. показывает оба значения;
3. указывает приоритетный источник по правилу;
4. отмечает влияние на KPI и прогноз;
5. запрашивает подтверждение руководителя.

## 10. Telegram

Для доставки создаётся отдельный Telegram-бот.

На сервере хранятся:

- токен бота;
- разрешённый `chat_id` руководителя;
- список разрешённых команд;
- журнал отправки и доставки;
- ссылки с ограниченным сроком действия на расширенный отчёт.

В Telegram не рекомендуется передавать полные персональные данные, договоры и платёжные реквизиты. В сообщении показываются агрегаты и безопасные идентификаторы; детализация открывается после авторизации в веб-интерфейсе.

## 11. Секреты, которые нельзя отправлять в чат или сохранять в репозитории

- полный URL входящего webhook Bitrix24;
- OAuth client secret;
- access token и refresh token;
- JSON-ключ сервисного аккаунта Google;
- приватный ключ Google;
- Telegram bot token;
- пароли и строки подключения к БД;
- персональные выгрузки инвесторов.

Все секреты передаются разработчику через утверждённый менеджер секретов или защищённый канал и устанавливаются как серверные переменные окружения.

## 12. Последовательность разработки интеграций

### Шаг 1. Подготовить тестовый доступ

- создать технического пользователя Bitrix24;
- назначить права просмотра;
- создать тестовый incoming webhook;
- создать сервисный аккаунт Google;
- выдать ему Viewer только на тестовую таблицу;
- создать тестового Telegram-бота.

### Шаг 2. Сделать инвентаризацию

- получить воронки, стадии и поля Bitrix24;
- получить структуру тестовых таблиц;
- составить карту полей;
- зафиксировать обязательные поля и типы.

### Шаг 3. Реализовать коннекторы

- Bitrix REST client с allowlist;
- Google Sheets reader с разрешёнными диапазонами;
- Telegram sender;
- повтор запросов, лимиты и журнал ошибок.

### Шаг 4. Сохранить исходные снимки

- схема базы данных;
- версии загрузок;
- защита персональных данных;
- контрольные суммы и аудит.

### Шаг 5. Нормализовать и сверить

- связать сделки, договоры и платежи по `deal_id`;
- проверить валюты, даты и суммы;
- сформировать отчёт ошибок качества.

### Шаг 6. Подключить дашборды

- сначала «Обзор», «Продажи», «Финансы» и «Данные»;
- затем «Команда», «Юристы», «Сопровождение» и «Прогноз»;
- каждый показатель раскрывается до источника.

### Шаг 7. Подключить ИИ-вывод

- передавать ИИ только проверенные аналитические факты;
- отделять расчёты от текстового объяснения;
- обезличивать данные согласно утверждённой политике;
- добавить ссылки на доказательства.

### Шаг 8. Пилот

- параллельно сравнивать с ручным отчётом;
- фиксировать расхождения;
- оценивать полезность выводов;
- не использовать агент как единственное основание критических решений до приёмки.

## 13. Критерии готовности подключения

- все разрешённые сущности Bitrix24 читаются без операций записи;
- структура каждой таблицы валидируется до загрузки;
- повторная загрузка не создаёт дубли;
- сделки, договоры и платежи связываются по стабильным ключам;
- каждый KPI воспроизводится из детализации;
- устаревший источник явно маркируется;
- конфликтующие данные не скрываются;
- секреты отсутствуют в клиентском коде, логах и репозитории;
- вечерний отчёт формируется из контрольного снимка и отправляется в установленное время;
- журнал подтверждает нулевое количество запросов записи в бизнес-источники.
